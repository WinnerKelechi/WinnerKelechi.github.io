Family Social Network System

Project Overview

Position: Lead Developer

Technology Stack: PHP, JavaScript, MySQL, Bootstrap, AJAX, JSON, HTML/CSS

The Family-Centric Social Network System is a web-based application designed to help families create and maintain a family tree while enabling communication among members through a chat platform...

Actors in the System

1. System Administrator (Family Tree Creator)

  • The first registered member who starts the family tree.
  • Receives a unique Family Identification Number (FIN) upon registration.
  • Adds other members and defines relationships.
  • Manages the family tree structure.

2. Members

  • Registered users who are related to the administrator by blood or marriage.
  • Can view and interact with the family tree.
  • Update their personal details and profiles.
  • Communicate with other family members via the chat platform.

Functional Requirements

A. Registration & Family Tree Management

  1. A family member (admin) registers to start the family tree.
  2. Admin receives a Family Identification Number (FIN).
  3. Admin can add new members and specify relationships.
  4. Members can access the tree using a unique identification number.
  5. Members can update their information.
  6. The system enables the proper tree structure creation by linking members correctly.

B. Communication & Social Networking

  • Chat Platform: Members can communicate in real time.
  • Members can send private messages to other family members.

C. Family Statistics & Insights

  • Display gender ratio (percentage of male to female members).
  • Show birthplace statistics (percentage of members born in different locations).
  • Generate a birth year scale (oldest to youngest member ranking).
  • Family Member Suggestions: System suggests potential family members based on surname and place of origin.

D. Profile Management

  • Members can update their profile information.
  • Profile images can be uploaded.
  • Relationship details can be edited or modified.

System Modules & Components

1. User Management Module

  • Admin & Member Registration
  • Profile Management
  • Login & Authentication

2. Family Tree Module

  • Dynamic family tree visualization
  • Relationship management (adding parents, siblings, children)

3. Chat Module

  • Private messaging between family members
  • Real-time chat feature

4. Analytics & Statistics Module

  • Gender ratio analysis
  • Birth year scale representation
  • Geographic distribution insights

5. Search & Suggestion Module

  • Search for family members using surname and place of origin
  • Suggest family connections based on existing data

System Architecture

Frontend

  • HTML5 / CSS3 / Bootstrap – For UI design and responsiveness.
  • JavaScript & AJAX – For dynamic user interactions.

Backend

  • PHP (Laravel / CodeIgniter) – For business logic and server-side processing.
  • MySQL – Database for storing user details, relationships, and messages.

Database Structure (MySQL Tables)

Table Name Description
users Stores user details (ID, name, email, role, FIN)
family_tree Stores relationships between members
messages Stores chat messages between members
statistics Stores family statistics (gender, birth locations, ages)

Security Considerations

  • Password Hashing (BCrypt) – Secure authentication.
  • Session Management – Prevent unauthorized access.
  • CSRF Protection – Secure data transmissions.

Tamara Store

Project Title:

Credit Card Fraud Detection on an E-Store (Tamara Store)

Position:

Lead Developer

Technology Stack:

  • PHP
  • JavaScript
  • MySQL
  • Bootstrap
  • AJAX
  • JSON
  • HTML/CSS

1. Project Overview

This project aims to develop a fraud detection system for Tamara Store using deep learning algorithms. The system will monitor credit card transactions in real-time, identifying fraudulent activities and minimizing false alarms.

2. Objectives

  • Implement a web-based fraud detection system.
  • Use deep learning algorithms (Autoencoder (AE) and Restricted Boltzmann Machine (RBM)) to detect fraudulent transactions.
  • Provide real-time notifications to administrators for suspicious transactions.
  • Enable quick blocking of compromised credit cards.
  • Improve fraud detection accuracy while minimizing false positives.

3. Advantages of the Proposed System

  • Faster fraud detection than existing methods.
  • Admin notifications for fraudulent activities.
  • Ability to block stolen cards immediately.
  • High accuracy in fraud detection.
  • Balanced prediction rates to minimize errors.

4. Functional Requirements

  • User Registration & Authentication: Users register with personal details and card information, including two-factor authentication.
  • Transaction Monitoring: Real-time tracking for fraudulent patterns.
  • Security Checks & Verification: Additional security questions for flagged transactions.
  • Fraud Handling: Recording suspicious transactions, notifying the admin, and blocking the card.
  • User Interface: Simple and intuitive UI using Bootstrap and JavaScript.

5. System Architecture & Flow

  1. User Registration: Consumers register and set up second-level security.
  2. Transaction Processing: Users place orders using credit cards, and transaction details are stored.
  3. Fraud Detection Mechanism:
    • AE analyzes past behavior and validates new transactions.
    • RBM processes transactions and reconstructs outputs.
    • If fraudulent, the system records and rejects the transaction.
  4. Fraud Handling & Admin Notification:
    • System notifies the admin to block the card.
    • SMS alerts notify the user about the blocked transaction.

Exam Portal

Position: Lead Developer

Technology Stack:

  • PHP
  • JavaScript
  • MySQL
  • Bootstrap
  • AJAX
  • JSON
  • HTML/CSS

1. Project Overview

This project aims to develop an Examination and Course Registration System to streamline course registration, scheduling, and exam writing processes in universities. The system will have two primary user roles: Administrators and Students.

2. System Users & Roles

Administrator Roles:

  • Examination Officer (Super-Admin)
    • Adds and manages lecturers.
    • Generates unique tokens for lecturers.
    • Provides login credentials to lecturers.
    • Oversees all system activities.
  • Head of Department (HOD)
    • Manages department-specific courses.
    • Approves student registrations.
    • Monitors examination processes.
  • Lecturers
    • Manage course details.
    • Upload examination questions.
    • Monitor student exam performance.

Student Roles:

  • Students register and enroll for courses.
  • They register for exams based on approved courses.
  • Students write exams online without location constraints.

3. System Modules & Functionalities

A. Administrator Panel

  • User Management:
    • Admin creates and manages lecturer accounts.
    • Unique tokens are generated for lecturer identification.
  • Course Management:
    • Admin can add, edit, and remove courses.
    • Assign courses to specific lecturers.
  • Exam Management:
    • Set examination schedules.
    • Manage exam content and questions.
    • Oversee grading and results processing.
  • Student Monitoring:
    • Track student registrations.
    • Approve or reject exam registrations.
    • View exam progress and performance analytics.

B. Student Panel

  • Registration & Login:
    • Students register via a dedicated form.
    • Login credentials are generated after registration approval.
  • Course Registration:
    • Select and enroll in available courses.
    • View registered courses and schedules.
  • Examination Module:
    • Register for upcoming examinations.
    • Take exams remotely with an anti-cheating mechanism.
    • View results after grading is completed.
  • Profile Management:
    • Update personal details.
    • View past exams and scores.

4. System Architecture

Frontend:

  • HTML/CSS, Bootstrap for UI design.
  • JavaScript, AJAX for dynamic content handling.

Backend:

  • PHP for server-side processing.
  • MySQL for database management.

Database Structure (MySQL)

Tables:

  • users (id, name, email, role, password, lecturer_token)
  • students (id, student_id, name, email, registered_courses)
  • courses (id, course_code, course_name, lecturer_id)
  • exams (id, course_id, exam_date, status)
  • questions (id, exam_id, question_text, options, correct_answer)
  • results (id, student_id, exam_id, score, status)
